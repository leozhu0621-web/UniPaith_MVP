# 70 ¬∑ Financial Fit & Direct Admission ‚Äî Scholarships, Net-Price, Reverse-Admissions

> The papers put a *price* and a *guarantee* on the match. Persuasion: *"recommend or provide a customized aid package within their means that can increase their appeal to each student ‚Ä¶ financial aid that matches the student's need, or a policy supplement for travel hardships"* (`Master Paper`:86) ‚Äî and every offer stays human-in-the-loop (`Master Paper`:88). The benchmark proves the category is real and won by reverse-admissions: **Niche Direct Admissions** put 1M+ students in front of a real no-application offer with a **$16,000 average scholarship** from 150+ partners (`Competition Analysis`:1722); **Appily Match** delivered 100k+ unsolicited admit+scholarship offers and ~**$6B** in scholarship value (`Competition Analysis`:328); **Common App Direct Admissions** reached 200+ institutions for 2025‚Äì26 (`Competition Analysis`:839); **Appily/Niche** also run a ~250,000-scholarship / ~$50M finder (`Competition Analysis`:325). `Feature_List_V1` names all four pieces: Probability Bands (L63), Net Price Estimator (L66), Scholarship Finder AI (L231), Financial Aid Calculator/EFC (L234).
>
> The substrate is unusually far along, and that is the point of this spec ‚Äî **operationalize, don't rebuild.** `60` shipped a real `Scholarship` table (`models/reference.py:43`: typed `eligibility` JSONB + `amount_min/max` + `distribution_history` + provenance) seeded with **4** rows (`services/crawler/seed.py:305`), but **no finder** ‚Äî only a flat list endpoint (`services/crawler/reference_service.py:187`). `NetPriceService` is a substantial, deterministic, tested estimator (`services/net_price_service.py:263,349`) that **already wires `ref_geo_cost`** into COA (`:434`) ‚Äî it computes net-cost ranges, affordability bands, and an `aid_scholarship_likelihood_band`, but it does **not** read the scholarship catalog and emits **no EFC**. `ai/probability.py:54` is *exactly* the line the roadmap flagged ‚Äî *"rule-based + calibrated heuristic over the historical admit rate"* (`:17`) ‚Äî emitting admit/scholarship/waitlist bands (`api/students.py:1098`), but it consumes only `acceptance_rate ¬∑ fitness ¬∑ confidence`, **not** `66` feasibility, `67` calibration, or `68` admit-history. And the **reverse-admissions / direct-admit engine does not exist at all**: institutions can release a per-applicant decision (`application_service.py:737` `release_decision` ‚Üí mints an `OfferLetter`, `models/application.py:264`), but there is no **standing eligibility rule** that proactively surfaces a guaranteed-admit + aid offer to a matching profile in discovery.
>
> **The distinction that defines this spec.** `34` is **reactive**: a human reviews *one* application that already arrived, then releases a decision. `39` collects the money once an offer exists. `70`'s reverse-admissions is the **proactive top-of-funnel**: an institution publishes a *standing* eligibility rule once (human-approved, audited), and any apply-ready student (`44`) who matches receives an auto-minted **guaranteed-admit + scholarship offer surfaced in discovery ‚Äî before they apply.** It reuses `34`'s offer-minting and `39`'s deposit rails downstream; it is not a second decision surface.
>
> Build anchor: extend `services/net_price_service.py` (read the catalog + emit EFC), `ai/probability.py` (consume `66`/`67`/`68`), `services/crawler/reference_service.py` (+ a new `ScholarshipFinderService`); new `services/direct_admit_service.py` + `eligibility_rules`/`direct_admit_offers` tables; reuse `application_service.release_decision` (`34`), `OfferLetter`, `AuditService` (`36`, `services/audit_service.py:99`), the apply-ready gate (`44`, `intake_engine_service.py:809`). Pairs with `60` (catalog), `66` (taste/feasibility), `67` (calibration), `68` (cost/outcome data), `46` ¬ß6/¬ß9 (fairness + consent), `40` (prospect audience), `39`/`34` (offers/money).
>
> Status: **draft v1.0** ¬∑ 2026-06-02 ¬∑ turns a real-but-listless scholarship table + a static probability heuristic into a finder + EFC + admit-probability engine, and adds the proactive reverse-admissions surface the benchmark is built on. Rule-based path stays the fallback (`tests/test_plan2_integration.py`).

## 2. Scholarship-Finder ‚Äî match awards to a profile (eligibility rules engine)

`60` gives a typed catalog (`Scholarship.eligibility` JSONB, `amount_min/max`, `is_renewable`, `deadline`, `distribution_history`, provenance) but no matcher. Build a **deterministic eligibility rules engine** ‚Äî the same posture as `net_price_service` (rule-based, unit-testable without a DB, honest-empty over fabrication).

- **New `services/scholarship_finder_service.py`** with a pure `match_scholarships(student_signals, scholarships) -> list[ScholarshipMatch]`. Each `eligibility` JSONB is a normalized predicate set over the **`42`/`44` signal vocabulary already in the profile** (intake registry, `services/intake/registry.py`): `field_of_study` (CIP/track), `degree_level`, `gpa_min`, `citizenship`/`residency`, `first_gen`, `demographic_openness`, `need_based`, `merit_band`, `geography`, `enrollment_status`. A scholarship matches when every present predicate is satisfied; missing student signals ‚Üí *eligible-pending* (asks for the field), never silently excluded.
- **Match output:** `{scholarship_id, status: eligible | eligible_pending | ineligible, award_estimate (amount_min..max), unmet_predicates[], provenance}` ‚Äî provenance carried straight from `Scholarship` (`source`/`fetched_at`/`confidence`) so the surface says "sourced from <domain>, updated N days ago" (`60` ¬ß4). Never present a crawled award as a commitment.
- **Surfacing.** A `GET /me/scholarships/match` endpoint (student) ranks eligible awards by `award_estimate √ó renewability √ó deadline-proximity`; feeds the net-price estimator (¬ß3) and the profile Financial tab. Institution/program-linked scholarships (`Scholarship.institution_id`/`program_id`) are matched per-program; external awards are profile-wide.
- **Fairness (`46` ¬ß6).** Eligibility predicates may encode *award-defined* criteria (e.g. a need-based or identity-specific scholarship the sponsor lawfully scopes) but the **finder's ranking never up/down-weights a student by a protected attribute** ‚Äî it surfaces every award the student is eligible for and lets the student choose. A predicate set that would let the *platform* infer a protected class for *ranking* is rejected by the same proxy-detection pass `66`/`67` use.
- **Flag** `ai_scholarship_finder_v2_enabled` (net-new, default off) gates an optional LLM **eligibility-extraction** pass that normalizes messy crawled eligibility prose into predicates (reuses the `60` `SourceExtractionAgent` / Qwen, eval-gated `62`, never invents a criterion absent from source); the rule-based predicate match is always the scoring path and the fallback.

## 3. Net-Price / EFC estimator ‚Äî extend the real one, add EFC

`net_price_service.py` already computes COA (tuition + fees + `ref_geo_cost` living + books + intl), an expected aid discount from `school_avg_net_price`/`net_price_by_income`/`68` `cost_data`, a `{min, expected, max}` range, an affordability band vs the student's budget, and `aid_scholarship_likelihood_band`. **Do not rebuild it.** Two extensions:

- **Wire the scholarship catalog in.** `_expected_discount` (`:175`) today knows nothing about ¬ß2 matches. Add a `matched_scholarships` input: the sum of *eligible* award estimates becomes an explicit aid line (capped at COA), shown as its own driver ("$X from N scholarships you're eligible for") distinct from the school's average-aid discount ‚Äî so the net-price range reflects awards the student can actually pursue, with provenance. Keep the range honest: scholarship aid is modeled as a **band** (eligible‚âÝawarded), widening `_UNCERTAINTY` (`:39`), never collapsing to a point.
- **EFC estimator (`Feature_List_V1`:234).** New pure `estimate_efc(household_income, household_size, assets, dependents, ‚Ä¶) -> {efc, methodology, range}` ‚Äî a simplified **Federal Methodology** approximation (income protection allowance + asset contribution), explicitly labeled an *estimate, not a FAFSA SAI* (the same `DISCLAIMER` posture, `:28`). `unmet_need = COA ‚àí EFC ‚àí expected_aid` feeds the affordability band and the ¬ß2 need-based predicate. Inputs come from the profile Financial tab (consented, `46` ¬ß9); absent income ‚Üí `available=False` with a reason (mirrors `_unavailable`, `:242`), never a fabricated EFC.
- **Stays deterministic.** No LLM in the math (the module's contract, `:9`). The estimator remains the rule-based path that the direct-admit aid-package (¬ß4) and probability scholarship band (¬ß5) read.

## 4. Reverse-admissions / direct-admit engine ‚Äî the proactive surface (net-new)

The benchmark's flagship and the papers' Persuasion step. **Standing eligibility rule** (published once, human-approved) ‚Üí **proactive guaranteed-admit + scholarship offer** to matching apply-ready profiles, surfaced in discovery. Explicitly **distinct from `34`** (per-applicant reactive decision on an arrived application) and from `66`'s recruitment *targeting* (which finds prospects to *nurture*; this *commits* an offer).

- **New `eligibility_rules` table** (one per program/intake, institution-authored): `program_id`, `status` (`draft | active | paused | archived`), predicate set in the **same `44` signal vocabulary** as ¬ß2 (`gpa_min`, `field_of_study`, `degree_level`, `test_band`, `residency`, ‚Ä¶ ), `scholarship_template` (award amount/type or a `Scholarship` link), `cap` (max offers this cycle), `valid_from/until`, `requires_human_publish` (default true), `created_by`, version (clone-and-modify like `student_strategies`). A rule is the institution's standing promise: *"any student matching this gets in, with this aid."*
- **Publishing is human-in-the-loop + audited (`Master Paper`:88, `36`).** A rule never goes `active` without an explicit institution-admin publish action; every publish/pause/edit writes an `AuditService` row (`services/audit_service.py:99`, category `direct_admit_rule`). Fairness-gated **before** activation: the predicate set passes `46` ¬ß6 proxy-detection + disparate-impact exactly as the `66` taste model does ‚Äî a rule that would direct-admit (or withhold) by a protected/proxy attribute is **blocked at publish**, surfaced on the fairness dashboard, and can trip `programs.matching_halted`.
- **Matching ‚Üí offer (new `services/direct_admit_service.py`).** A job (and an inline check when a profile reaches apply-ready) evaluates active rules against students who: (a) pass the rule's predicates, (b) are **apply-ready** for that program (`intake_engine_service.get_apply_ready`, `44`), and (c) consented to proactive matching (`46` ¬ß9). On match, under the `cap`, it **reuses `34`** ‚Äî `application_service.release_decision(..., decision="accepted"|"conditional_admission", offer={scholarship_amount, conditions, response_deadline})` mints the existing `OfferLetter` (`models/application.py:264`), with a new `direct_admit_offers` provenance row linking offer‚Üîrule‚Üîstudent for audit and cap accounting. **No new offer schema** ‚Äî direct-admit is a *source* of the same offer, flagged `offer_source='direct_admit'`.
- **Surfaced in discovery, not the inbox.** The student sees a proactive **"You've been offered admission"** card in `/s` discovery / `/s/explore` (this is the one earned celebratory beat on the student side), carrying the net-price (¬ß3) and scholarship (¬ß2) context so the offer is legible. Accepting routes through the existing offer-response flow; any deposit routes to `39` `PaymentService` (no new money code). Declining/expiry frees the cap.
- **The aid package (`Master Paper`:86).** The `scholarship_template` is the "customized aid package within their means" ‚Äî sized against the student's EFC/unmet-need (¬ß3) where the rule opts into need-aware sizing, so the offer is *persuasive* (covers the gap) rather than flat. Need-aware sizing is itself fairness-bounded (sizes to *need*, never to a protected attribute).
- **Flag** `ai_direct_admit_v2_enabled` (net-new, default off). When off, rules can be authored and stored but **no auto-offer is minted** (the engine is dark); the human per-applicant path (`34`) is unaffected. An optional LLM pass only drafts the *offer message copy* (reuses `OutcomeBriefForOfferLetter`, `45` ¬ß15) ‚Äî never the eligibility decision, which is deterministic.

## 5. Admit-probability bands ‚Äî make `ai/probability.py` consume real signals

`ai/probability.py:54` is the flagged static heuristic: it lifts the program's `acceptance_rate` by `fitness` and widens by `(1‚àíconfidence)` to emit admit/scholarship/waitlist bands ‚Äî but it never sees `66` feasibility, `67` calibration, or `68` admit-history. Extend it to consume them while **keeping the exact rule-based fallback**.

- **Feasibility from `66`.** Add an optional `feasibility` input (the `65` ¬ß4 / `66` ¬ß3 `admit_feasibility(student, program)` term ‚Äî how close the student sits to the program's admit distribution). When present, it *replaces* the crude `0.5 + fitness` lift (`:100`) with a learned proximity; absent (cold-start) ‚Üí the current heuristic lift, unchanged. The shape the function emits is identical (the module's substitution contract, `:18`), so callers (`api/students.py:1098,1253`) don't change.
- **Admit-history from `68`.** Today `acceptance_rate` is the program's single published rate. `68` provides typed admit-rate / yield / class-profile trends; feed a band-specific historical rate (per round/cohort where available) instead of one scalar ‚Äî narrower, more honest bands. Still `None` when there is no history (`:91`), preserving the "Not enough data yet" honesty.
- **Calibration from `67`.** Route the emitted band centers through `confidence_calibrator.py` once `67` fits it on ‚â•1,000 confidence-outcome pairs; until then bands flow **uncalibrated and labeled as such** (truthful-low beats false precision, `64` ¬ß7). The `MATCH_READY_MIN_CONFIDENCE` gate (`:27`) stays.
- **Scholarship band ‚Üê ¬ß2.** Today the scholarship band is a fit-gated guess (`:108`). Where the ¬ß2 finder returns eligible institution/merit awards (or a ¬ß4 direct-admit rule applies), the scholarship band tightens to reflect *real* eligible aid rather than a heuristic ‚Äî and a live direct-admit offer makes it a near-certainty.
- **Bands feed the direct-admit surface and the explore dual-score** (`Feature_List_V1`:63) ‚Äî the same `MatchResult.probability_bands` shape (`schemas/matching.py:55`). `66`/`70` own the probability model; `65` consumes it as the `admit_feasibility` *feature* (per `65` open-Q recommendation).

## 6. How it fits the engine (no duplication)

- `65` reads `66`'s `admit_feasibility` as a fitness term; `70` owns the *probability bands* that present it to the student. One model, two surfaces.
- `34` stays the per-applicant reactive decision/offer surface; `70` is the proactive standing-rule surface that *reuses* `34`'s mint. `39` stays the money layer both feed.
- `60` owns crawling + provenance for the scholarship catalog; `70` consumes it (finder + net-price) and never re-crawls. `68` owns typed cost/outcome data; `70` reads it.
- `66` owns the institution taste/feasibility/segmentation; `70`'s eligibility rules are a *narrower, deterministic, human-published* commitment layered on top of (not replacing) taste-based targeting.

## 8. Acceptance

- [ ] A student with a populated profile gets ranked **eligible** scholarships from the catalog with award estimates and provenance; a missing-signal award shows `eligible_pending` (asks for the field), never a false exclude.
- [ ] The net-price estimate reflects matched eligible awards as a distinct, sourced aid line; an EFC + unmet-need is returned when income is provided and **honestly unavailable** when it isn't.
- [ ] An institution can author a standing eligibility rule; it produces **no offer** until a human publishes it, and a rule predicated on a protected/proxy attribute is **blocked at publish** (`46` ¬ß6 test) ‚Äî distinct from `34`'s per-applicant path.
- [ ] A matching, apply-ready (`44`), consented (`46` ¬ß9) student receives an auto-minted direct-admit offer reusing `34`'s `OfferLetter` (no new offer schema), capped per cycle, fully audited (`36`); accept routes to `39` for any deposit.
- [ ] `ai/probability.py` consumes `66` feasibility / `68` history / `67` calibration when available and returns the **exact current bands** when they aren't (`ai_probability_bands_enabled` off ‚Üí existing behavior).
- [ ] Disabling `ai_direct_admit_v2_enabled` / `ai_scholarship_finder_v2_enabled` returns the current rule-based behavior; no model path 5xxs.

## 9. Open questions

- **EFC methodology depth** ‚Äî full FM/IM replica vs the simplified income-protection approximation. *Recommend simplified-and-labeled for v1 (an estimate, not an SAI); never imply a real FAFSA result.*
- **Direct-admit cap exhaustion** ‚Äî first-come vs ranked-by-fit when more students match than the cap allows. *Recommend ranked by fitness within fairness bounds (highest-fit first), waitlist the remainder for the next cap window.*
- **Need-aware aid sizing** ‚Äî does the offer size to the student's unmet need (¬ß3), and is that fair? *Recommend opt-in per rule, sized to need not to any protected attribute, fairness-gated like every other path.*
- **Scholarship eligibility-extraction trust** ‚Äî auto-apply LLM-parsed predicates vs review. *Recommend review-gated at first (like `60` enrichment), loosen on measured extraction precision (`62`).*
- **Offer expiry ‚Üí re-offer** ‚Äî if a student declines a direct-admit offer, can the same rule re-offer next cycle? *Recommend no auto-re-offer within a cycle; freed cap goes to the next match.*
