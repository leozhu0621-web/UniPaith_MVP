Design notes

How it works

  • Living profile = warm reflection of the same signals the backend already extracts (interests/personality, goals, needs) — just relabeled human ("What lights you up / Where you're headed / What you need to thrive"), not Profile/Goals/Needs/layers.
  • Everything editable (your spec's editable-inference rule); edits feed back into matching.
  • Gaps shown as gentle invitations ("Uni could understand you better if we talk about budget →"), never a progress bar or "incomplete."
  • Handoff is counselor-led: when signal is strong enough, Uni offers matches in-thread; no "reach 50% on 3 tracks."
  • Always-available: student can request matches anytime; Uni shows them with an honest confidence note.

Under the hood (keeps your backend)

  • Conversation still produces goals / needs / identity signals via the existing discovery extractor → the matching engine is unchanged.
  • "Strong enough to offer matches" = the existing readiness/handoff signal, surfaced as Uni's judgment instead of a % gate.
  • "See programs that fit me" → the existing strategy/match flow (Explore), so nothing downstream breaks.
  • Old tracks/layers/strategy-CTA UI is removed from Discover; the data model stays.
